## 1. 数学运算——深度学习世界的"原子"

### 1.1 七种基本二元运算

两个张量逐元素做运算——这是所有复杂操作的**最小可分解单元**。

| 操作 | 公式 | 在 LLM 中干啥 |
|------|------|-------------|
| **Add** | a + b | 残差连接（"跳跃"）、偏置加法 |
| **Sub** | a − b | 残差计算（"和原来差了多少"） |
| **Mul** | a × b | **门控的灵魂**——SwiGLU 的 gate×up、注意力权重×Value |
| **Div** | a / b | 归一化中的除法、缩放 |
| **Pow** | a^b | RoPE 的频率计算 |
| **Max** | max(a, b) | 最大池化 |
| **Min** | min(a, b) | 截断、Clip |

> 💡 **Mul 是门控机制的基石**。从 LSTM 的遗忘门 (`f_t ⊙ c_{t-1}`) 到 SwiGLU 的 `Swish(gate) ⊙ up`，所有"选择性放行"的操作都建立在逐元素乘法之上。一个操作数充当**门控信号**（0~1 范围），另一个充当**被控信息**。理解了这一点，GLU 家族的所有变体就不神秘了——它们只是在"门控信号的生成方式"上做文章。

### 1.2 广播 (Broadcasting)——张量的"自动对齐术"

两个张量形状不完全一样时，框架会自动"扩展"较小的张量。这件事你天天在用，但你真的理解规则吗？

**规则只有一句话**：从最后一个维度向前比较，每个维度要么相等，要么其中一个为 1。

```
张量 A: [B, C, 1, 1]    ← C 个标量，每个在 (1,1) 的空间位置上
张量 B: [B, 1, H, W]    ← 一张 H×W 的图，通道维度是 1

广播后: [B, C, H, W]    ← 标量被"涂满"整张图，图被"复制"到每个通道

逐维度检查:
  轴0: B vs B → 相等 ✓
  轴1: C vs 1 → 1 扩展为 C ✓
  轴2: 1 vs H → 1 扩展为 H ✓
  轴3: 1 vs W → 1 扩展为 W ✓
```

**不能广播的情况**：`[3, 4] + [2, 4]` → 轴0 是 3 vs 2，都不为 1 → ❌

> 💡 在深度学习中，BN 的 γ/β（[C]）广播到 [B,C,H,W]、注意力掩码（[seq,seq]）广播到 [batch,heads,seq,seq]——这些都是广播在默默工作。你不需要手动 `expand`，框架帮你做了。

### 1.3 一元运算与归约运算

**一元运算**（每个元素独立变换）：

| 操作 | 公式 | 在哪出现 |
|------|------|---------|
| **Exp** | eˣ | Softmax 的核心 |
| **Sqrt** | √x | RMSNorm 的均方根 |
| **Log** | ln(x) | 交叉熵损失 |
| **Floor/Round** | ⌊x⌋ / round(x) | **量化取整** |

**归约运算**（多个值合并为一个）：

| 操作 | 公式 | 本质 |
|------|------|------|
| **Sum** | Σxᵢ | Softmax 的分母 |
| **Mean** | Σxᵢ/n | GlobalAvgPool |
| **Max** | max(xᵢ) | Softmax 的数值稳定 |
| **L2Norm** | √(Σxᵢ²) | 权重衰减 |

> 💡 **归约是"从多到少"的核心操作**。这也是为什么 GPU 上的归约比 Intel CPU 复杂——归约要求线程间通信，而 GPU 的几千个线程并不擅长互相等待。高效的 GPU 归约用的是**树形通信**（tree reduction），O(log n) 而非 O(n) 的串行步骤。

### 1.4 MatMul / GEMM / InnerProduct——三位一体的矩阵大师

这是神经网络参数化的核心——80%+ 的计算量都在这里。

```
MatMul:      C = A × B              最通用的矩阵乘法
GEMM:        C = α·A·B + β·C        加上了缩放和累加
InnerProduct: y = Wx + b            GEMM 的特例——封装了权重和偏置
```

**三者的关系**：InnerProduct ⊂ GEMM ⊂ MatMul。几乎每一个带可学习参数的操作最终都落到矩阵乘法上——卷积用 im2col 变成 MatMul、Attention 的 QKV 投影是 MatMul、MLP 那三层是 MatMul、最后的 Projection 也是 MatMul。

### 1.5 EinSum——用一行字符表达任意张量运算

EinSum 是一种用字符串描述张量缩并的"微型语言"。编译器会自动选择最优的计算顺序。

```python
# 矩阵乘法: [M,K] × [K,N] → [M,N]
torch.einsum('ij,jk->ik', A, B)

# Attention Q·K^T: [B,H,Q,D] × [B,H,K,D] → [B,H,Q,K]
torch.einsum('bhqd,bhkd->bhqk', Q, K)

# 逐通道缩放 (SE Block): [B,C,1,1] × [B,C,H,W] → [B,C,H,W]
torch.einsum('bc,bchw->bchw', scale, x)
```

**规则速查**：消去重复字母 → 保留未出现字母。`'ij,jk->ik'` 中 j 重复了→消去，i 和 k 保留→输出 ik。

> 💡 现代框架会把 EinSum 自动分解为最优的底层算子组合（BatchMatMul + 逐元素乘 + 归约）。你只需要写出数学意图，优化交给编译器。

## 2. 激活函数——给网络注入"灵魂"

### 2.1 为什么要非线性？

故事是这样的：如果没有激活函数，无论你堆叠多少层线性变换，从输入到输出永远只是一个线性变换——`W_n(...W_2(W_1·x)) = W_effective·x`。100 层 = 1 层。激活函数的非线性打破了这种"层数的幻象"，让深层网络真正能拟合复杂函数。

**反向传播角度的另一层理解**：如果激活函数的梯度恒为 0（如 ReLU 的负值区），反向传播到此为止，前面的参数再也无法更新。这就是"死亡 ReLU"的根源。一个好的激活函数必须**在大部分输入范围上有非零梯度**。

### 2.2 进化路线——一场从"硬截断"到"软门控"的进化史

```
ReLU (2012):        max(0, x)                         ← 硬截断, 死亡区
  ↓
LeakyReLU/PReLU:    x>0?x:αx                          ← 负值区开了一条"小缝"
  ↓
ELU/SELU:           x>0?x:α(eˣ-1)                     ← 负值平滑, 自归一化
  ↓
GELU (2018):        x·Φ(x)                            ← 概率门控, BERT标准
  ↓
Swish/SiLU (2017):  x·σ(x)                            ← 自门控, LLaMA标准
  ↓
SwiGLU (2022):      Swish(xW₁) ⊙ (xW₂) · W₃           ← 显式门控, 现代LLM标配
```

### 2.3 核心激活函数详解

#### ReLU——最经典的简单粗暴

```
ReLU(x) = max(0, x)

优点: 一个 max 操作搞定，计算快到令人发指
缺点: x≤0 时梯度为 0 → "死亡 ReLU"
     一旦某个神经元的所有输入都 ≤0，它就永远死了——梯度过不来

直觉: 就像一个只有"开"和"关"两档的开关。
     开到"关"时，信息彻底切断，连梯度都传不回去。
```

#### GELU——BERT 的选择

```
GELU(x) = x · Φ(x)   (Φ 是标准正态分布的累积分布函数)

直觉: 不像 ReLU 的问"x>0？"（非黑即白），
     GELU 问的是"x>0 的概率有多大？"
     当 x=0:  Φ(0)=0.5 → GELU=0·0.5=0      "不确定，暂时关掉"
     当 x=2:  Φ(2)=0.98 → GELU=2·0.98=1.96  "很确定，几乎全放行"
     
     这是一种"概率门控"——模型根据自己对输入的"信心"来决定放行多少。
```

#### Swish/SiLU——现代 LLM 的标配

```
Swish(x) = x · σ(x) = x / (1 + e⁻ˣ)

直觉: x 同时扮演了两个角色——
     "信息本身" 和 "自己的门控信号"
     x 很大 → σ(x)≈1 → Swish≈x        "有话说，大声说"
     x≈0   → σ(x)≈0.5 → Swish≈0.5x   "不太确定，小声点"
     x 很负 → σ(x)≈0 → Swish≈0        "不对，闭嘴"

与 GELU 极其相似，但计算更简单（Sigmoid 比标准正态 CDF 好算）。
代表: LLaMA, Qwen, Mistral——当前所有主流开源 LLM。
```

#### SwiGLU——门控 FFN 的新范式

```
SwiGLU(x) = Swish(xW_gate) ⊙ (xW_up) · W_down

为什么中间维度是 3072 (3×1024) 而不是 4096 (4×1024)？
  标准 FFN (ReLU):   W1: 1024×4096, W2: 4096×1024 → 2×4=8 个参数块
  SwiGLU:            gate: 1024×3072, up: 1024×3072, down: 3072×1024 → 3×3=9 个参数块
  → 两者的总参数量接近（8≈9），但 SwiGLU 多了一层门控 → 精度更好

直觉: gate 分支产生 0~1 的软门控信号，up 分支携带信息。
      gate⊙up = "对 up 的每个维度，根据 gate 的置信度决定放行多少"
      → 模型学会了"选择性表达"
```

#### Sigmoid vs Softmax——一对容易混淆的兄弟

| 对比 | Sigmoid | Softmax |
|------|---------|---------|
| **决策方式** | 逐元素独立判断 | 整个向量联合决策 |
| **多分类** | 各类独立（可以多个激活） | **互斥选择**（概率和为 1） |
| **典型场景** | 二分类、门控信号 | 多分类、注意力权重 |
| **输出特征** | 每个值独立在 (0,1) 中 | 所有值之和 = 1 |

> 💡 一句话区分：Sigmoid 是"这个样本是不是猫？"（独立判断）。Softmax 是"这张图是猫还是狗还是鸟？"（联合决策，只能选一个）。

### 2.4 选哪个？——实战速查

| 场景 | 推荐 | 为什么 |
|------|------|--------|
| CNN 分类 | ReLU / GELU | 简单够用 |
| Transformer (BERT系) | GELU | 训练稳定 |
| 现代 LLM MLP | **SwiGLU** | 当前最优 |
| 移动端部署 | HardSwish | 无 exp 运算，快 |
| LSTM/GRU 门控 | Sigmoid | 天然 0~1 输出 |

## 3. 归一化——让训练不再"震荡"

### 3.1 问题：为什么需要归一化？

训练深层网络时，每一层输出的分布都在不断漂移。前面层更新了参数，后面层接收到的输入分布就变了——就像射击训练中靶子在不停移动，射手需要不断重新适应。这导致：
- 需要更小的学习率（慢）
- 需要更谨慎的初始化（烦）
- 深层网络容易梯度消失（痛）

归一化相当于**把靶子钉死在同一个位置**——每一层的输入都被拉到均值 0、方差 1 附近，后面层可以稳定学习。

### 3.2 五种归一化的核心差异——一张图看懂

```
             N (batch)    C (channel)   H,W (spatial)

BatchNorm:  ──────────   ✗ 做归一化    ✗ 做归一化   ← 跨 batch 统计，CNN 专用
LayerNorm:  ✗ 每个独立   ───────────  ───────────  ← 跨 feature 统计，Transformer 专用
RMSNorm:    ✗ 每个独立   ───────────  ───────────  ← LayerNorm 的精简版
GroupNorm:  ✗ 每个独立   分组内做      ───────────  ← BN 和 LN 的折中
InstanceN:  ✗ 每个独立   每通道独立    ───────────  ← 风格迁移专用
```

### 3.3 BatchNorm——CNN 时代的王者

```
训练:  y = (x − μ_batch) / √(σ²_batch + ε) · γ + β
推理:  y = (x − running_mean) / √(running_var + ε) · γ + β
       ↑ 推理时用训练累积的统计量，不依赖 batch

致命弱点: batch size 小了效果就差
         Transformer 的 batch size 通常不大 → BatchNorm 不适用
```

**推理时的一个魔法：Conv-BN 融合**

BN 在推理时是一个线性变换（running_mean/running_var 是固定的）：

```
BN(x) = γ × (x − μ) / √(σ² + ε) + β
      = (γ / √(σ²+ε)) × x + (β − γ·μ / √(σ²+ε))
      = α × x + b'

融合进 Conv:
  ConvBN(x) = α × (W·x + b) + b' = (αW)·x + (αb + b')
  
→ 在模型加载时一次性算好新的 W' 和 b'
→ 推理时完全无 BN 开销！
```

### 3.4 RMSNorm——现代 LLM 的共同选择 💡

```
LayerNorm:  y = (x − mean(x)) / √(var(x) + ε) · γ + β
RMSNorm:    y = x / √(mean(x²) + ε) · γ

精简了什么？
  - 均值减法 (x − mean)      → 去掉！省一次归约
  - 偏置项 β                 → 去掉！省一半参数

代价呢？精度损失 < 0.1%，几乎可以忽略。

为什么现代 LLM（LLaMA/Qwen/Mistral）全部选 RMSNorm？
  28 层模型:
    每层省 1 次归约求 mean + 1 次减法 + 1 次加法 + 1 个可学习参数向量的存储
    28 层累计 → 省 ~30% 归一化计算量 + 50% 归一化参数量
```

## 4. 硬件执行模型——为什么你的算子可能"看起来快，实际慢"

### 4.1 一个反直觉的事实

```
为什么 [1024,1024] 的矩阵乘法 (1M FLOPs) 可能比 [1024,1024] 的加法 (0.001M FLOPs) 快？

加法:
  计算量: 1024² = 1M 次加法 ≈ 0.001ms (CPU)
  内存访问: 读 4MB + 写 4MB ≈ 0.1ms
  → 内存访问占了 99% 的时间！

MatMul:
  计算量: 2×1024×4096×1024 ≈ 8.6G FLOPs ≈ 8ms
  内存访问: 读 16MB + 写 4MB ≈ 0.3ms
  → 计算占了 99% 的时间！

结论: 算得慢不可怕，等的慢才可怕。
```

### 4.2 Roofline Model——判断瓶颈的神器

```
                    算力天花板 (TFLOPS)
                    ┃
     计算密集区      ┃        ← 瓶颈在算力: MatMul, 大Conv
         ╲          ┃
          ╲         ┃
  ─────────╲────────┃──────  ← 内存带宽天花板
             ╲       ┃
   内存密集区  ╲      ┃        ← 瓶颈在带宽: Add, ReLU, RMSNorm, Decode
                ╲    ┃
                 ────┸────→ 计算强度 (FLOPs/Byte)

计算强度 = 总FLOPs / 总内存访问字节数

roofline 拐点 (A100): ~150 FLOPs/Byte
  < 150: memory-bound — 优化布局, 融合, 压缩
  > 150: compute-bound — 优化算法, 更大batch, TensorCore
```

### 4.3 SIMD——"一次打包 16 个"

```
标量: for i in range(N): c[i] = a[i] + b[i]     → N 次运算
SIMD: c[0:16] = a[0:16] + b[0:16]               → 1 次运算！

不同架构的宽度:
  ARM NEON:  4 个 FP32 / 一次
  x86 AVX2:  8 个 FP32 / 一次
  x86 AVX-512: 16 个 FP32 / 一次
  CUDA warp: 32 个线程 / 一次

ncnn 的做法: 每个算子写 4 个版本
  - 通用 C++ (fallback)
  - ARM NEON (手机)
  - x86 AVX2 (PC)
  - Vulkan shader (跨平台 GPU)
```
